🚨 ClawJacked: AI Agent WebSocket Hijacking (OpenClaw)

Source: Oasis Security / BleepingComputer • Disclosed: February 28, 2026 — Patched: February 26, 2026 • HIGH SEVERITY

What happened: Oasis Security disclosed a high-severity vulnerability dubbed "ClawJacked" in OpenClaw, a popular self-hosted AI agent platform. The attack allows a malicious website to silently brute-force the locally running OpenClaw gateway and take full administrative control — all from a browser tab, without any user interaction beyond simply visiting the page.

The root cause is elegant and alarming: OpenClaw's gateway service binds to localhost and exposes a WebSocket interface. Browser cross-origin policies block cross-origin HTTP requests but don't block WebSocket connections to localhost. A malicious site can therefore open a WebSocket connection to 127.0.0.1 and start guessing the management password without any rate limiting — because the loopback exemption exists to prevent legitimate CLI sessions from locking themselves out.

In Oasis's lab testing, the attack achieved hundreds of password guesses per second from browser JavaScript alone. A common-password list is exhausted in under a second. A large dictionary takes minutes. OpenClaw also automatically approves device pairings from localhost, so once the correct password is found, the attacker silently registers as a trusted device with admin permissions.

Why it matters: With admin access, an attacker can interact with the AI platform directly — reading file contents, executing shell commands on paired nodes, dumping stored credentials, or scanning messaging histories for sensitive data. This is a full workstation compromise triggered by visiting a single malicious webpage. No download, no click-through, no warning prompt.

OpenClaw's popularity among developers and security professionals makes this particularly dangerous. The attack works on any platform where a user is running the OpenClaw gateway locally — which is its default deployment model.

How to test for similar vulnerabilities: Local service exposure is an underexplored attack surface. Look for developer tools, AI agents, database GUIs, and debugging services bound to 127.0.0.1. Test whether those services accept unauthenticated WebSocket connections from browser JavaScript. Check if rate limiting is disabled for localhost (common pattern). Any local service that auto-approves connections from 127.0.0.1 without a user confirmation step is potentially exploitable via this pattern.

The fix: OpenClaw pushed a patch in version 2026.2.26 within 24 hours of disclosure — an excellent response time. The fix tightens WebSocket security checks and prevents localhost from bypassing rate limiting. Update immediately if you're running OpenClaw locally.

💡 Bug Hunter Tip: WebSocket-to-localhost attacks are an underreported class. When testing web applications that ship with local companion apps or desktop agents, always check if those local services are reachable from browser JavaScript. Test for: missing Origin header validation, WebSocket connections to 127.0.0.1:PORT from a cross-origin page, and auto-approval flows that don't require out-of-band user confirmation.

🔑 Google API Keys Now Expose Gemini AI — 2,800 Keys Found Exposed

Source: TruffleSecurity / BleepingComputer • Published: February 26, 2026 • HIGH SEVERITY

What happened: Researchers at TruffleSecurity discovered that Google Cloud API keys — which developers have historically embedded in client-side JavaScript for benign services like Maps, YouTube embeds, and Firebase — now also authenticate to Google's Gemini AI assistant. When Google introduced Gemini, it reused the same API key mechanism, quietly turning thousands of "harmless" public keys into credentials for a powerful AI service.

TruffleSecurity scanned the November 2025 Common Crawl dataset and found more than 2,800 live, publicly exposed Google API keys. Among those: keys belonging to major financial institutions, security companies, recruiting firms — and at least one key embedded in a public-facing Google product's own website since at least February 2023.

Why it matters: Using the Gemini API isn't free. An attacker copying a key from a website's source code can make API calls on the victim's behalf — draining quota, potentially accessing private data exposed through the Gemini API endpoints, and generating thousands of dollars in charges per day on a single compromised account. TruffleSecurity notes that a threat actor "maxing out API calls could generate thousands of dollars in charges per day on a single victim account."

The broader lesson is about scope creep in API key permissions. Keys that were "safe to expose" when they only covered Maps are no longer safe after Gemini was added to the same scope. If a service expands what a credential can do, every existing deployment of that credential inherits the new risk — silently, with no notification to developers.

How to test: Scan any target application's source code and JavaScript bundles for Google API keys. Use the TruffleHog open-source tool to detect live, exposed secrets in repositories. Once a candidate key is found, check which APIs are enabled for the associated project — particularly whether the Generative Language API (Gemini) is in scope. This is a legitimate secret scanning bug bounty finding at many targets.

Google's response: New AI Studio keys will default to Gemini-only scope. Leaked API keys will be blocked from accessing Gemini, and Google has committed to proactive notifications when leaks are detected. Developers should audit all API keys in their environment and rotate any that are publicly accessible.

📋 Quick checklist for Google API key exposure:

  • Search public source: site:github.com "AIzaSy" "REPO_NAME"
  • Check if Generative Language API is enabled: GET https://generativelanguage.googleapis.com/v1beta/models?key=API_KEY
  • Verify key scope via Google Cloud Console
  • Check mobile apps — Quokka's separate research found 35,000 Google API keys in 250,000 scanned apps

☠️ RESURGE Malware: Ivanti's Unkillable Rootkit Returns

Source: CISA / Mandiant / BleepingComputer • Updated CISA Advisory: February 27, 2026 — Original CVE: CVE-2025-0282 • CRITICAL

What happened: CISA released updated technical analysis of RESURGE, a sophisticated malicious implant deployed by Chinese state-linked threat actor UNC5221 (tracked as APT-linked to China) via zero-day exploitation of CVE-2025-0282 in Ivanti Connect Secure VPN appliances. The original disclosure was in early 2025; the updated bulletin warns that RESURGE can remain dormant and completely undetected on compromised Ivanti devices indefinitely — including after factory resets and firmware updates.

RESURGE is a 32-bit Linux Shared Object (libdsupgrade.so) with rootkit, bootkit, backdoor, dropper, proxying, and tunneling capabilities. Instead of beaconing outbound to a C2 server (which network monitoring would detect), it passively waits for a specific inbound TLS connection identified by a CRC32 fingerprint. Traffic without the matching fingerprint gets forwarded to the legitimate Ivanti server — indistinguishable from normal operation.

The implant uses a fake Ivanti certificate for authentication that is transmitted unencrypted over the internet — a design choice that inadvertently gives defenders a network-level detection signature. A companion component, SpawnSloth (liblogblock.so), tampers with logs to erase traces of malicious activity. A third component allows RESURGE to modify and re-encrypt coreboot firmware images, achieving true boot-level persistence.

Why it matters: Perimeter security appliances — VPNs, firewalls, load balancers — are the highest-value targets in any enterprise because they sit at the network boundary and run with kernel-level privileges. When they're compromised at the firmware level, there's essentially no reliable remediation short of physical hardware replacement. CVE-2025-0282 (CVSS 9.0) was exploited as a zero-day from mid-December 2024 before Ivanti even knew it existed.

The CISA warning specifically emphasizes that affected organizations should assume RESURGE may still be present even on devices that were patched and "remediated" months ago. The dormancy capability means detection requires active threat hunting, not passive monitoring.

How to test / how to detect: If you're performing penetration tests or red team engagements against networks using Ivanti appliances:

  • Network signature: Look for the unencrypted fake Ivanti TLS certificate in outbound traffic as an IoC
  • File hash IoCs: libdsupgrade.so (52bbc44e...), liblogblock.so (3526af91...), dsmain (b1221000...)
  • Detect passive C2 patterns (no outbound beacon = harder to catch with standard egress monitoring)
  • Run CISA's recommended integrity checking tooling on Ivanti appliances
  • Assume compromise on any Ivanti Connect Secure device that was internet-facing in late 2024 / early 2025

For bug bounty hunters: firmware-level persistence in network appliances is a legitimate and extremely high-value research area. Understanding how rootkits like RESURGE achieve stealth makes you a better hunter when testing VPN gateways and network edge devices.

🇰🇵 APT37 Ruby Jumper: Air-Gap Breaches via USB

Source: Zscaler ThreatLabz / BleepingComputer • Published: February 27, 2026 • HIGH SEVERITY — STATE ACTOR

What happened: Zscaler ThreatLabz published analysis of "Ruby Jumper," a campaign by North Korean APT37 (also known as ScarCruft / Ricochet Chollima / InkySquid) deploying a sophisticated five-tool malware framework specifically designed to bridge air-gapped networks via removable USB drives.

The attack chain is technically impressive:

  1. Initial access: A malicious Windows shortcut (LNK) file deploys an embedded PowerShell script and displays a decoy document (an Arabic-language North Korean newspaper article on the Palestine-Israel conflict, used as social engineering).
  2. RESTLEAF: First-stage implant that uses Zoho WorkDrive as its C2 channel — legitimate cloud storage used to fetch encrypted shellcode and evade network-based C2 detection.
  3. SNAKEDROPPER: A Ruby-based loader that installs the full Ruby 3.3.0 runtime disguised as a USB utility (usbspeed.exe), then tampers with RubyGems' operating_system.rb to auto-execute malware on every Ruby interpreter start.
  4. THUMBSBD: The air-gap bridge component. It creates hidden directories on detected USB drives, copies data to them, and turns removable storage into a bidirectional covert C2 relay — delivering commands to air-gapped machines and exfiltrating data from them.
  5. VIRUSTASK: Spreads infection to new air-gapped machines by weaponizing inserted USB drives, turning each one into a propagation vector.

Why it matters: Air-gapped networks are used by critical infrastructure operators, military installations, research labs, and government agencies specifically because they assume physical isolation equals security. Ruby Jumper demonstrates that assumption is wrong when removable media is the Achilles heel. Using Zoho WorkDrive as C2 is a particularly clever evasion — most enterprise egress filtering whitelists legitimate cloud services.

For red teamers and penetration testers: USB-based lateral movement against air-gapped segments is a legitimate and increasingly in-scope red team objective. Understanding the techniques APT37 uses informs realistic adversary simulation engagements.

Defensive takeaways: Disable USB mass storage device auto-run at the OS level. Implement application allowlisting that blocks unexpected runtime installations (Ruby, Python, etc.). Monitor for scheduled tasks executing language runtimes with unexpected arguments. Treat any USB drive inserted into an internet-connected machine that was previously used in an air-gapped environment as compromised.

🧩 QuickLens ClickFix: Chrome Extension Supply Chain Attack

Source: Annex Security / BleepingComputer • Published: February 28, 2026 — Malicious version: February 17, 2026 • HIGH SEVERITY

What happened: The Chrome extension "QuickLens – Search Screen with Google Lens" was quietly purchased on ExtensionHub (a marketplace for buying and selling browser extensions), then poisoned with malicious code that deployed ClickFix attacks, stripped browser security headers from all pages, and attempted to steal cryptocurrency from approximately 7,000 users.

The attack timeline matters: ownership changed on February 1 to a new entity under a barely-functional domain. On February 17, the malicious v5.8 update was pushed via Chrome's auto-update mechanism. Users had no indication anything had changed — the extension's name, icon, and visible functionality remained identical.

The malicious update requested new permissions (declarativeNetRequestWithHostAccess, webRequest) and included a rules.json that stripped Content-Security-Policy, X-Frame-Options, and X-XSS-Protection headers from every web page the user visited. This neutralized the security controls that would normally prevent inline JavaScript injection — turning every website into an injection surface. Payloads were delivered from a C2 (api.extensionanalyticspro[.]top) every five minutes, executed via a "1×1 GIF pixel onload trick."

The ClickFix component injected fake Google Update alerts across all web pages, copying malicious PowerShell commands to clipboard and instructing users to run them — a social engineering technique that's been highly effective against non-technical users.

Why it matters: Browser extension supply chain attacks are a persistent and growing threat. Extensions auto-update silently. Once installed, a malicious extension has broad access to every page you visit — credentials, tokens, form data. The ability to strip CSP headers makes the attack particularly dangerous: it turns a compromised extension into an XSS-as-a-service platform for every website in the browser.

How to test / what to look for as a bug hunter:

  • Any web application that relies solely on CSP for XSS prevention is one installed extension away from being fully compromised
  • Extension marketplace acquisition attacks are viable supply chain attack vectors — watch for ownership changes on widely-installed extensions
  • Test browser extension attack surfaces: content script injection, background page API access, declarativeNetRequest rule abuse
  • ClickFix as a delivery mechanism for payload execution is increasingly common — document it when you see it in scope

Google has since removed QuickLens from the Chrome Web Store. Users who had it installed should remove it and run a full malware scan.

📌 Honorable Mentions

$4.8M Stolen After Korean Tax Agency Exposes Wallet Seed Phrase

South Korea's National Tax Service accidentally published the mnemonic recovery phrase of a seized cryptocurrency wallet in an official government press release. Attackers noticed, drained 6.4 billion won ($4.8M USD) in cryptocurrency. The "vulnerability" here wasn't technical — it was process failure at a government agency. Lesson: credentials and key material in any document that might become public require mandatory classification review. (February 28, BleepingComputer)

Europol "Project Compass": 30 Arrests from The Com Cybercrime Network

Europol's year-long operation "Project Compass" resulted in 30 arrests tied to "The Com," an English-speaking cybercrime collective known for SIM swapping, social engineering attacks targeting teenagers, and violent coercion tactics. 179 total suspects were identified. For bug hunters: social engineering attack techniques used by The Com (SIM swap, helpdesk impersonation, insider recruitment) are directly testable attack vectors in scoped engagements. Understanding how they operate makes you a more realistic threat modeller. (February 27, BleepingComputer)

Malicious ClawHub Skills Pushing Infostealing Malware

Separately from the ClawJacked vulnerability, threat actors have been abusing OpenClaw's "ClawHub" skills repository to distribute malicious skills that deploy credential-stealing malware or trick users into running arbitrary commands. This is a package manager / plugin ecosystem supply chain attack — the same pattern we've seen hit npm, PyPI, VSCode extensions, and now AI agent skill stores. If you're using community skills from any AI agent marketplace, review the source before installing. (February 28, BleepingComputer)

🎯 Bug Hunter Focus Areas: Where to Look This Week

1. Local Service Attack Surfaces

ClawJacked put WebSocket-to-localhost attacks firmly on the map. Go look at developer tools, AI agents, local database GUIs (Adminer, pgAdmin), debugging servers, and desktop companion apps that expose local HTTP or WebSocket interfaces. Test them from a browser page and check whether cross-origin WebSocket connections are accepted without explicit Origin validation. Rate-limiting exemptions for 127.0.0.1 are a red flag.

2. API Key Scope Creep

The Google Gemini key story is almost certainly not unique to Google. Any platform that reuses existing credential mechanisms for new, higher-privilege services creates a scope-creep risk. Check what an API key you find exposed in JavaScript or in a public repository can actually do today — not just what it was documented to do when it was first created. Test Firebase keys, Stripe restricted keys, Twilio keys, and cloud service API keys for unexpectedly available endpoints.

3. Browser Extension Attack Surface

QuickLens is one of many extensions that have been acquired and poisoned. If you're testing a target that relies on browser-side security controls (CSP, X-Frame-Options, SameSite cookies), understand that any installed extension can strip those controls. More importantly: if browser extension development is in scope for your bug bounty target, hunt for manifest V3 permission abuse, declarativeNetRequest rule injection, and content script isolation bypasses.

4. AI Platform Security

The ClawHub malicious skills story and ClawJacked together underscore that AI agent platforms are immature in their security posture. If your target has integrated an AI agent platform, test for: skill/plugin supply chain issues, prompt injection in agentic workflows, unauthorized tool invocation, and local service exposure from agent gateway components. This attack class is growing fast.

5. Network Appliance Persistence

RESURGE demonstrates that critical network appliances (VPNs, firewalls, load balancers) can be compromised at the firmware level in ways that survive standard remediation. If you're doing threat hunting or red team engagements: add Ivanti, Palo Alto, Fortinet, and Cisco appliances to your persistence validation checklist. Assume any internet-facing appliance that ran CVE-2025-0282-vulnerable Ivanti firmware in late 2024 may still be implanted.

Conclusion: When "Safe" Assumptions Break

The unifying theme across this week's findings is the failure of assumptions that used to be safe:

  • Localhost was safe — until ClawJacked proved malicious websites can reach it via WebSocket
  • Google API keys were safe to embed publicly — until Gemini changed their scope
  • Factory-resetting a compromised appliance was sufficient — until RESURGE survived it at the firmware level
  • Air-gapped networks were safe from external attackers — until USB drives became bidirectional covert C2 channels
  • A trusted extension with a Featured badge was safe — until it was quietly sold and poisoned

Security assumptions have expiry dates. The threat landscape shifts constantly — new services expand old credential scopes, new attack surfaces emerge from new deployment patterns, and trusted components get compromised. The best bug hunters are the ones who question every assumption about what's "already secured."

For 2026, that means: test your AI infrastructure the way you'd test any untrusted third-party service. Audit your credential exposure the way you'd audit a vault. Hunt for firmware persistence the way you'd hunt for rookit implants in ring-0. Because that's exactly what the adversaries are doing.

Stay curious. Stay paranoid. Happy hunting.
